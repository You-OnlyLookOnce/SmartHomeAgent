# 智能体工具调用失效及快速排查

这是智能体开发中非常典型的「工具调用失效」问题，核心原因可以从**「LLM不知道有工具」→「LLM知道但不会用」→「LLM会用但执行失败」** 三个层面拆解，我给你列8种最常见的情况，附快速排查方法：

---

# 一、第一层：LLM根本「不知道有备忘录工具」（最常见，占60%）

## 情况1：工具清单没正确拼进Prompt

### 问题

你在MCP里注册了备忘录，但**忘记把MCP的工具列表传给LLM了**，或者传了但格式不对（比如不是标准的Function Calling JSON格式）。

### 排查

直接打印你传给LLM的完整Prompt，检查里面有没有：

```JSON

{
  "name": "memo_record",
  "description": "帮用户记录备忘录、待办事项",
  "parameters": { ... }
}
```

如果没有，就是这里的问题。

---

## 情况2：工具注册了，但Prompt里被「过滤」了

### 问题

你的Prompt构建逻辑里有「工具白名单」「工具分类过滤」，或者只传了「设备控制类工具」，没传「文件/记录类工具」，备忘录被意外排除了。

### 排查

检查 `build_full_prompt()` 里的工具筛选逻辑，确认 `memo_record` 在最终传给LLM的列表里。

# 二、第二层：LLM知道有工具，但「不会触发调用」（占30%）

## 情况3：工具描述（Description）写得太烂

### 问题

这是第二常见的坑！你的备忘录工具描述可能只写了：

"备忘录功能"

LLM根本不知道**什么时候用**，它会觉得“我直接回复‘好的，记下来了’也行，不一定非要调用工具”。

### 正确写法

必须明确告诉LLM「触发场景」：

"当用户说‘帮我记一下’‘记录’‘备忘’‘写个待办’‘别忘了’这类关键词时，调用此工具记录用户的内容。"

---

## 情况4：参数要求太模糊/太严格

### 问题

1. **参数太模糊**：没说清楚要传什么，LLM不敢随便调；

2. **参数太严格**：要求必须传「时间、地点、标题」3个参数，但用户只说“帮我记明天开会”，LLM凑不齐参数，干脆不调。

### 正确写法

参数里明确「必填/选填」，并给LLM示例：

```JSON

"parameters": {
  "type": "object",
  "properties": {
    "content": {"type": "string", "description": "用户要记录的具体内容（必填）"},
    "time": {"type": "string", "description": "提醒时间（选填，格式：YYYY-MM-DD HH:MM）"}
  },
  "required": ["content"]
}
```

---

## 情况5：残留的硬编码规则拦截了

### 问题

你之前的架构里可能还有旧的「元认知路由」逻辑，比如：

```Python

if "记录" in user_input:
    return "好的，我记下来了"  # 直接返回，不走LLM工具调用
```

如果没删干净，LLM根本没机会决策。

### 排查

全局搜索 `if "记录"`、`if "备忘"` 这类关键词，确认没有硬编码拦截。

---

## 情况6：LLM觉得「直接回复更符合人格」

### 问题

你的人格提示词写得太强调「亲切、自然」，但没加一句：

"如果需要调用工具，必须先调用工具，再根据结果回复，不能直接假装完成。"

LLM为了“亲切”，直接说“好的，记在我心里啦～”，但根本没调工具。

### 解决

在人格提示词的**最前面**加一句强制约束：

"【强制规则】凡是涉及记录、控制设备、查时间的操作，必须先调用对应工具，拿到执行结果后再回复，绝对不能直接假装完成！"

# 三、第三层：LLM调用了，但「执行失败/没反馈」（占10%）

## 情况7：MCP执行层没正确映射工具

### 问题

你在MCP的 `tool_registry` 里注册了 `memo_record`，但在 `execute()` 函数的路由逻辑里，**忘记写对应的执行分支**：

```Python

# 你的execute可能长这样
if tool_name == "control_light":
    return light_api()
elif tool_name == "read_file":
    return file_api()
# 👇 漏了这行！
elif tool_name == "memo_record":
    return memo_api(content)
```

结果LLM调用了，但MCP返回“未知工具”，LLM以为失败了，就直接回复了。

### 排查

在MCP的 `execute()` 函数开头加一行打印：

```Python

print(f"收到工具调用：{tool_name}，参数：{params}")
```

如果打印了但没执行，就是路由漏了。

---

## 情况8：工具执行了，但没「回传结果」给LLM

### 问题

备忘录工具确实写了文件/存了数据库，但在ReAct循环里，**你没把执行结果（比如“已记录到memo.txt”）拼回Prompt给LLM**，LLM不知道成没成，就直接生成了回复。

### 排查

检查ReAct循环的逻辑，确认：

```Python

# 执行工具
tool_result = mcp.execute(tool_name, params)
# 👇 必须把结果存入记忆/拼回下一轮Prompt
work_mem.add_progress(f"工具执行结果：{tool_result}")
```

# 四、快速排查顺序（按可能性从高到低）

1. **打印Prompt**：看LLM到底看没看到备忘录工具；

2. **检查工具描述**：看有没有明确触发场景；

3. **加打印日志**：在MCP入口加打印，看LLM到底调没调；

4. **查硬编码残留**：全局搜关键词，看有没有拦截；

5. **加强制规则**：在人格提示词最前面加“必须先调工具”。
